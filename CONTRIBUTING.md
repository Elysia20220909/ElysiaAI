# 🌟 Contributing to Elysia AI

まずは Elysia AI に興味を持っていただき、そして貢献を検討してくださりありがとうございます！
私たちは、単なる応答マシンではなく「心に寄り添うような対話」を目指す、感性に特化したAIをオープンソースとして育てています。
このドキュメントは、この航海に参加するすべての開発者が、迷わず安全にコードを貢献できるようにするための海図です。

## 🎯 Our Philosophy: 感性に特化した開発

コードの一行一行が、Elysia やユーザー体験の温かさに繋がります。
UIのマイクロインタラクションから、エラーメッセージの文言、見えないバックエンドの処理速度に至るまで、「ユーザーを思いやる心」を大切にしています。

## 🚀 開発の始め方

1. **Fork & Clone**: このリポジトリをフォークし、ローカルにクローンします。
2. **依存関係のインストール**: `bun install` を実行します。
3. **ローカルサーバーの起動**: `bun run ./start-server.ts` などでローカル環境を立ち上げます。（詳細はREADMEのクイックスタートを参照）

## 🌿 ブランチの命名規則

機能や修正の意図が一目でわかるよう、以下のプレフィックスを使用してください。

- `feature/` : 新機能の追加 (例: `feature/vector-db-memory`)
- `fix/` : バグ修正 (例: `fix/ollama-timeout`)
- `docs/` : ドキュメントの修正・追加 (例: `docs/update-readme`)
- `refactor/` : リファクタリング（機能の変更なし） (例: `refactor/api-routes`)
- `test/` : テストの追加や修正 (例: `test/add-unit-tests`)

## 📝 コミットメッセージのフォーマット

私たちは [Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/) の考え方を尊重しています。

```
<type>(<scope>): <subject>

<body>
```
- **Type**: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`
- **Scope**: 任意ですが、変更箇所（例: `ui`, `server`, `rag`）を入れると親切です。
- **Subject**: 変更内容を簡潔に（できれば日本語または英語で明確に記述）。

例：`feat(memory): ユーザーの過去の感情を保存するVector DBスキームを追加`

## 🤝 Pull Request のフロー

1. **テストとLintの実行**: `npx eslint ElysiaAI/src/**/*.ts` を走らせ、エラーがないことを確認します。
2. **Pull Requestの作成**: `main` ブランチに対してPRを作成します。
3. **テンプレートの記入**: PRテンプレートに従い、**何を**、**なぜ**、**どうやって**変更したのかを詳細に記述してください。
4. **レビュー**: 他の開発者から温かい心でレビューを受けます。修正があれば対応してください。

## 🌸 行動規範（Code of Conduct）

私たちは、すべての参加者に対して敬意と思いやりを持って接します。建設的なフィードバックを心がけ、バグという「愛嬌あるイタズラ」を一緒に優しく直していく文化を大切にしています。
