# メンテナー

このドキュメントは、ElysiaAI プロジェクトのメンテナーと所有権移転プロセスを説明します。

## 現在のメンテナー

### プロジェクトオーナー
- **[@chloeamethyst](https://github.com/chloeamethyst)** - プロジェクト創設者・リードメンテナー
  - すべてのコンポーネントの責任者
  - 最終的な意思決定権限
  - セキュリティ問題の主要連絡先

## メンテナーの責任

### コードレビュー
- すべてのPRを適切にレビューし、品質基準を維持
- コードスタイル、設計パターン、セキュリティのベストプラクティスを確認
- 建設的なフィードバックとガイダンスを提供

### Issue管理
- 新しいIssueにラベルとマイルストーンを割り当て
- バグレポートと機能リクエストをトリアージ
- コミュニティの質問に対応

### リリース管理
- バージョニング戦略に従う（Semantic Versioning）
- CHANGELOGを維持
- リリースノートを作成
- タグとリリースを作成

### セキュリティ
- セキュリティ脆弱性を責任を持って処理
- セキュリティアップデートを適時にリリース
- セキュリティベストプラクティスを実施

### コミュニティ
- 行動規範を実施
- 健全で歓迎的な環境を育成
- 貢献者を指導・支援

## メンテナーになる方法

プロジェクトへの継続的で質の高い貢献を示した貢献者は、メンテナーに招待される可能性があります。

### 期待される貢献
- 複数の高品質なPRのマージ
- コードレビューへの積極的な参加
- Issue管理とコミュニティサポート
- ドキュメントの改善
- プロジェクトの長期的なビジョンへの関与

### 招待プロセス
1. 既存のメンテナーが候補者を推薦
2. すべてのメンテナーで議論
3. コンセンサスが得られたら招待を送付
4. 候補者が受諾したらCODEOWNERSとこのドキュメントを更新

## 所有権移転ガイド

### 新しいメンテナーの追加

```bash
# 1. GitHubリポジトリ設定でコラボレーターを追加
# Settings > Collaborators and teams > Add people

# 2. 適切な権限を付与
# - Read: 読み取りのみ
# - Triage: Issueとディスカッションの管理
# - Write: コミットとPRのマージ
# - Maintain: リポジトリ設定の一部管理
# - Admin: 完全な管理権限

# 3. CODEOWNERSファイルを更新
# .github/CODEOWNERS に新しいメンテナーを追加

# 4. このドキュメント（MAINTAINERS.md）を更新

# 5. チームに通知
```

### プロジェクト所有権の完全移転

プロジェクトの所有権を別の個人または組織に移転する場合：

#### ステップ1: 準備
```bash
# すべての変更がコミットされていることを確認
git status

# すべてのブランチをプッシュ
git push origin --all

# すべてのタグをプッシュ
git push origin --tags
```

#### ステップ2: ドキュメント更新
- [ ] LICENSE の著作権情報を更新
- [ ] README.md のメンテナー情報を更新
- [ ] MAINTAINERS.md（このファイル）を更新
- [ ] package.json の author, repository を更新
- [ ] CODEOWNERS を更新

#### ステップ3: GitHub設定
1. GitHub リポジトリ設定へ移動
2. "Settings" > "General" > "Danger Zone"
3. "Transfer ownership" をクリック
4. 新しいオーナーのユーザー名を入力
5. リポジトリ名を確認
6. 移転を実行

#### ステップ4: 移転後の確認
- [ ] 新しいオーナーが管理アクセス権を持つことを確認
- [ ] CI/CD シークレットが移転されたことを確認
- [ ] 外部サービス連携（Vercel、Herokuなど）を更新
- [ ] ドメイン設定を更新（該当する場合）
- [ ] npm/パッケージレジストリの所有権を更新（該当する場合）

### セキュリティ資格情報の管理

移転時に必ず以下を処理：

```bash
# 新しい JWT シークレットを生成
openssl rand -base64 32

# 新しい暗号化キーを生成
openssl rand -base64 32

# 新しい API キーをすべてのサービスで生成
# - Ollama API
# - Redis（本番環境）
# - データベース（本番環境）
# - 外部API統合
```

#### GitHub Secrets の更新
1. Settings > Secrets and variables > Actions
2. すべてのシークレットを削除または更新
3. 新しいオーナーが新しいシークレットを追加

## 意思決定プロセス

### 軽微な変更
- 単一のメンテナーがレビュー・承認可能
- 例: バグ修正、ドキュメント更新、リファクタリング

### 中程度の変更
- 2人以上のメンテナーのレビューが必要
- 例: 新機能、APIの変更、依存関係の更新

### 重大な変更
- すべてのメンテナーのレビューとコンセンサスが必要
- 例: アーキテクチャの変更、ライセンス変更、セキュリティポリシー

## 非アクティブなメンテナーのポリシー

6ヶ月以上活動がないメンテナーは：
1. 他のメンテナーから連絡を受ける
2. 継続する意思がない場合、名誉メンテナーとして記録
3. アクティブなメンテナーリストから削除

## 緊急連絡先

セキュリティ問題やクリティカルな問題の場合：
- GitHub Issues（公開情報の場合）
- セキュリティ脆弱性: [SECURITY.md](docs/SECURITY.md) を参照
- プライベート連絡: リポジトリのSecurity Advisoriesを使用
