# プロジェクトガバナンス

## 概要

ElysiaAI はオープンソースプロジェクトであり、コミュニティ主導の開発モデルを採用しています。このドキュメントは、プロジェクトの意思決定プロセスと運営構造を説明します。

## プロジェクト構造

### プロジェクトオーナー
- 最終的な意思決定権限を持つ
- セキュリティとインフラストラクチャの責任者
- プロジェクトの方向性を決定

### メンテナー
- コードレビューとマージ権限
- Issue とPRの管理
- リリースプロセスの実行
- 詳細は [MAINTAINERS.md](MAINTAINERS.md) を参照

### 貢献者
- コードやドキュメントを提供
- Issue報告と機能提案
- コミュニティサポート
- 詳細は [CONTRIBUTORS.md](CONTRIBUTORS.md) を参照

## 意思決定プロセス

### レベル1: 日常的な変更
**例**: バグ修正、タイポ修正、小さなリファクタリング

**プロセス**:
1. PRを作成
2. 1人のメンテナーがレビュー
3. 承認後マージ

**タイムライン**: 通常24-48時間

### レベル2: 機能追加
**例**: 新しいエンドポイント、UI改善、パフォーマンス最適化

**プロセス**:
1. Issue で提案とディスカッション
2. 設計の合意
3. PRを作成
4. 2人以上のメンテナーがレビュー
5. 承認後マージ

**タイムライン**: 通常1週間

### レベル3: 重大な変更
**例**: アーキテクチャ変更、APIの破壊的変更、セキュリティポリシー

**プロセス**:
1. RFC（Request for Comments）をIssueで作成
2. 少なくとも2週間のコミュニティディスカッション
3. すべてのメンテナーのレビューとコンセンサス
4. 実装計画の承認
5. PRの作成とレビュー
6. すべてのメンテナーの承認後マージ

**タイムライン**: 通常1ヶ月以上

## RFC (Request for Comments) プロセス

大きな変更には RFC が必要です。

### RFC を作成すべき変更
- 新しい主要機能
- APIの破壊的変更
- アーキテクチャの大幅な変更
- セキュリティポリシーの変更
- ライセンスの変更

### RFC テンプレート

```markdown
# RFC: [タイトル]

## 概要
[変更の簡潔な説明]

## 動機
[なぜこの変更が必要か]

## 詳細設計
[技術的な詳細]

## 代替案
[検討した他のアプローチ]

## 未解決の問題
[まだ決まっていないこと]

## 影響
- 既存機能への影響
- パフォーマンスへの影響
- セキュリティへの影響
- 互換性への影響
```

### RFC のライフサイクル
1. **提案**: Issue として作成
2. **ディスカッション**: コミュニティからのフィードバック
3. **修正**: 必要に応じて RFC を更新
4. **投票**: メンテナーによる投票
5. **承認/却下**: 最終決定
6. **実装**: 承認された場合、PRを作成

## リリースプロセス

### バージョニング
[Semantic Versioning 2.0.0](https://semver.org/) に従います：

- **MAJOR**: 破壊的変更
- **MINOR**: 後方互換性のある機能追加
- **PATCH**: 後方互換性のあるバグ修正

### リリーススケジュール
- **PATCH**: 必要に応じて（バグ修正）
- **MINOR**: 月次（新機能）
- **MAJOR**: 必要に応じて（破壊的変更）

### リリースプロセス
1. CHANGELOGを更新
2. バージョン番号を更新（package.json）
3. タグを作成（`git tag v1.2.3`）
4. GitHubでリリースを作成
5. リリースノートを公開
6. コミュニティに通知

## コードレビューガイドライン

### レビュアーの責任
- [ ] コードが要件を満たしているか
- [ ] コードスタイルガイドに従っているか
- [ ] テストが含まれているか
- [ ] ドキュメントが更新されているか
- [ ] セキュリティの問題がないか
- [ ] パフォーマンスへの影響を考慮したか

### レビュープロセス
1. **初回レビュー**: 24-48時間以内
2. **フィードバック**: 建設的で具体的に
3. **変更要求**: 明確な理由と提案を提供
4. **承認**: すべての懸念が解決されたら

### マージ基準
- [ ] すべての CI チェックがパス
- [ ] 必要な数のメンテナーが承認
- [ ] すべてのコメントが解決
- [ ] コンフリクトが解消
- [ ] CHANGELOG が更新（該当する場合）

## 行動規範の実施

[CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md) 違反の報告は：

1. プライベートにメンテナーに連絡
2. メンテナーが調査を実施
3. 適切な措置を決定
4. 報告者に結果を通知

### 措置のレベル
1. **警告**: 初回の軽微な違反
2. **一時停止**: 繰り返しまたは中程度の違反
3. **永久禁止**: 重大または継続的な違反

## コミュニティ参加

### ディスカッション
- GitHub Discussions: 一般的な議論、質問、アイデア
- GitHub Issues: バグレポート、機能リクエスト
- Pull Requests: コード貢献

### コミュニケーションチャンネル
- **GitHub Issues**: 技術的な議論
- **GitHub Discussions**: コミュニティQ&A
- **Pull Requests**: コードレビュー

## セキュリティ

セキュリティ脆弱性の報告は [SECURITY.md](docs/SECURITY.md) を参照してください。

### セキュリティ開示プロセス
1. 脆弱性を非公開で報告
2. メンテナーが確認（48時間以内）
3. 修正を開発
4. セキュリティアップデートをリリース
5. 公開開示（修正リリース後）
